### 왜 플로이드 워셜 알고리즘을 사용했나요?

고객마다 출발지와 도착지가 다르고 업체 위치도 여러 개라서 매번 최단거리를 구하면 비효율적입니다.  
도시 개수가 최대 30개여서 플로이드 워셜로 모든 도시 간 최단거리를 미리 구하는 방식으로 진행했습니다.

### 시간복잡도는?

`도시 수 = N,  도로 수 = R,  차량 수 = C, 고객 수 = Q, 한 도시의 차량 갯수 = M` 일때, 시간복잡도는
플로이드 워샬 : O(N^3)
차량 정렬 :O(C log C)
고객 처리 : O(Q * N * log M)

최단거리 전처리는 O(N^3), 각 고객마다 모든 업체 도시를 돌고 차량은 이분 탐색을 하므로 Q(QxNxlog M) 입니다.

### 왜 차량을 capacity 기준으로 정렬 했나요?

고객의 짐 무게 이상을 실을 수 있는 가장 작은 capacity 차량을 빠르게 찾기 위해 정렬했습니다. 이후 이분 탐색으로 O(logM)에 찾습니다.  


### 적재량이 더 큰 차가 요금이 더 저렴하면 어떻게 되나요?

현재 코드는 capacity 기준 첫 번째 차량을 선택하므로 rate 단조 증가 조건이 없으면 틀릴 수 있습니다. 그 경우 capacity 이상인 차량들 중 최소 rate를 찾도록 suffix minimum을 만들어야 합니다.


### 동률 처리 기준은?

sortedCities를 사전순으로 돌기 때문에 결과적으로 가격이 같으면 사전순 빠른 도시가 유지됩니다.  

### INF = Long.MAX_VALUE / 4 이유는?

최단거리 계산에서 dist[i][k] + dist[k][j]를 하므로 오버플로우를 피하려고 충분히 큰 값으로 줄였습니다.  

### 왜 long을 썼나요?

거리 합과 요금 곱셈에서 값이 커질 수 있어서 오버플로우 방지를 위해 long을 사용했습니다
